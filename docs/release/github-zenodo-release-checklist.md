# GitHub + Zenodo release checklist

This project is set up to support a frozen archival release through GitHub and
Zenodo.

Files already prepared in the repository root:

- `.zenodo.json`
- `CITATION.cff`

Important note:

- Zenodo uses `.zenodo.json` in preference to `CITATION.cff` when both files are
  present.
- GitHub uses `CITATION.cff` for the "Cite this repository" panel.

## Recommended release target

Suggested first archival release:

- tag: `v1.0-pilot`
- GitHub release title: `v1.0-pilot`

## What to freeze before release

1. Code state
- ensure the Unity runtime, offline preprocessing, and Phase 3 analytics reflect
  the exact version described in the manuscript
- confirm the repository branch is clean

2. Data and outputs
- decide whether the release will contain:
  - code only
  - code + anonymised pilot outputs
  - code + anonymised pilot outputs + supplementary figures/tables
- if participant-level exports are not going public, keep only the derived
  analytics outputs and document the restriction in the paper

3. Metadata decisions still required
- final software/data licence
- whether anonymised participant-level CSV exports are public or available on
  request
- final public DOI/URL to insert into the paper

## Zenodo setup

1. Sign in to Zenodo.
2. Open the GitHub integration settings in Zenodo.
3. Authorise GitHub if prompted.
4. Enable archiving for:
   - `MadaoSurmanito/aoi360-eye-tracking-system`

## Final repository checks before tagging

1. Review `.zenodo.json`
- update title/description only if the release scope changed
- if a final licence has been decided, also add it in the Zenodo deposit or the
  metadata file

2. Review `CITATION.cff`
- if you want the GitHub citation panel to show a release version, add:
  - `version: "v1.0-pilot"`
  - `date-released: YYYY-MM-DD`

3. Review the release contents
- source code archive
- optional release assets such as:
  - `pilot-analytics-outputs.zip`
  - `phase3-html-viewer.zip`
  - `paper-figures-and-tables.zip`
  - `anonymized-pilot-data.zip` if allowed

## GitHub release steps

1. Create the tag:

```powershell
git tag v1.0-pilot
git push origin v1.0-pilot
```

2. Create a GitHub release for `v1.0-pilot`.
3. Upload any additional release assets.
4. Publish the release.

## Zenodo DOI steps

1. Wait for Zenodo to ingest the GitHub release.
2. Open the generated Zenodo record.
3. Review and complete the metadata:
- creators
- description
- keywords
- licence
- access conditions, if any

4. Publish the Zenodo record if it remains in draft.
5. Copy the version DOI generated by Zenodo.

## After Zenodo publication

1. Insert the DOI into the manuscript `Data Availability` statement.
2. If needed, update `CITATION.cff` with the exact release version and date.
3. Optionally create a follow-up patch release if you only need metadata cleanup.

## Suggested `Data Availability` wording

### If anonymised participant outputs are public

```tex
\subsection*{Data Availability}
The source code, generated AOI assets, anonymised participant-level outputs, and
derived analytics tables required to reproduce the results reported in this
manuscript are archived at Zenodo:
\url{https://doi.org/10.5281/zenodo.XXXXXXX}.
The corresponding development repository is available at
\url{https://github.com/MadaoSurmanito/aoi360-eye-tracking-system}.
```

### If participant-level outputs remain restricted

```tex
\subsection*{Data Availability}
The source code and reproducible software artefacts are archived at Zenodo:
\url{https://doi.org/10.5281/zenodo.XXXXXXX}, with the development repository
available at \url{https://github.com/MadaoSurmanito/aoi360-eye-tracking-system}.
Due to participant privacy and data-protection constraints, the anonymised
fixation-level pilot exports are not publicly released in full. Derived analytics
outputs and additional anonymised materials are available from the corresponding
author on reasonable request.
```
